# RTB 2.0 — Complete Redesign Spec

## Overview

RTB 2.0 is a complete rewrite of the Remote Terminal Bridge, transforming it from a Node.js terminal access tool into a Rust-based hybrid platform that is both a general-purpose remote terminal and a first-class AI Agent management system.

The redesign incorporates the best patterns from two reference projects:
- **Mitto** (Go) — Observer pattern, sequence number tracking, two-tier caching, security hardening, comprehensive testing
- **VibeAround** (Rust) — Event-driven hub architecture, trait-based abstractions, Arc+DashMap concurrency, GPU-accelerated terminal rendering, plugin system

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Core positioning | Hybrid (terminal + AI Agent) | General remote terminal with first-class ACP Agent support |
| Backend | Rust (Axum) | Performance, memory safety, native single-binary distribution |
| Frontend | React 19 + Vite + Tailwind + shadcn/ui | Largest ecosystem, best AI-assisted coding support |
| Desktop/Mobile | Tauri 2 (including Tauri Mobile for iOS/Android) | Rust unified stack, cross-platform, small binary |
| Architecture | Unified Daemon (single binary) | Simplest deployment, zero config, proven by VibeAround |
| IM integration | Full plugin architecture (all IM via JSON-RPC) | Cleanest design, no built-in bias, any language |
| Agent protocol | ACP only | Standard protocol, Claude Code and Gemini CLI already support it |
| Tunnel | Pluggable (Cloudflare/Ngrok/Localtunnel as plugins) | Same plugin infrastructure as IM |
| User model | Single user, single instance | Personal tool, avoid complexity |

### Crate Dependency Rules

- **core** depends only on tokio runtime and basic libraries. No Axum or Tauri dependency. Can be tested independently.
- **server** is a thin I/O adapter layer. Axum handlers only do protocol conversion (HTTP/WS <-> core types). All business logic lives in core.
- **plugin-host** manages plugin subprocess lifecycles. Communicates via stdin/stdout JSON-RPC. Plugin crashes don't affect the main process.
- **cli** and **tauri-app** both depend on server + core. They are alternative entry points.
- Web frontend is compiled into the server binary via rust-embed. Zero file dependencies in production.

## §2 — Event Bus & Data Flow

### Central Event Bus

All components communicate through a central event bus based on `tokio::broadcast::channel` (capacity: 256 events, type: `Arc<BusEvent>`).

Subscribers: PTY Hub (pub+sub), Agent Hub (pub+sub), Server (sub -> WS), IM Bridge (pub+sub), Session Store (sub -> persist).

### BusEvent Enum

```rust
pub enum BusEvent {
    // PTY events
    PtyOutput     { session_id, seq, data: Bytes },
    PtyExited     { session_id, exit_code },
    PtyCreated    { session_id, name, shell },
    PtyResized    { session_id, cols, rows },

    // Agent events
    AgentMessage  { session_id, seq, content: AgentContent },
    AgentStatus   { session_id, status: AgentStatus },
    AgentToolUse  { session_id, tool, input, result },
    AgentError    { session_id, error, class: ErrorClass },

    // Session events
    SessionCreated  { session_id, session_type },
    SessionDeleted  { session_id },
    SessionSwitched { session_id },

    // System events
    TunnelReady   { url: String },
    TunnelDown    { reason: String },
    PluginLoaded  { plugin_id, name },
    PluginError   { plugin_id, error },
}
```

### Core Data Flow Scenarios

**Scenario 1: Browser terminal input**
```
Browser --ws--> Server (ws_terminal) --write--> PTY Hub
                                                  |
                                            pty stdout
                                                  |
                                                  v
                                             Event Bus
                                         PtyOutput{data}
                                                  |
                              +-------------------+-------------------+
                              v                   v                   v
                        Server              Session Store         IM Bridge
                        ws -> browser       append .jsonl         -> feishu/tg
```

**Scenario 2: ACP Agent conversation**
```
Browser --ws--> Server (ws_agent) --send--> Agent Hub
                                              |
                                      ACP process stdout
                                              |
                                              v
                                         Event Bus
                                 AgentMessage{content, seq}
                                              |
                              +---------------+----------------+
                              v               v                v
                        Server          Session Store      IM Bridge
                        ws -> browser   persist events     throttle -> IM
```

**Scenario 3: IM message to terminal**
```
Feishu Plugin --jsonrpc--> IM Bridge --> Event Bus
                                            |
                               route by session_id
                                            |
                               +------------+------------+
                               v                         v
                         PTY Hub                   Agent Hub
                         write to stdin            send ACP message
```

### Key Design Points

- **Sequence numbers (seq)**: Per-session AtomicU64, used for event ordering, deduplication, and incremental sync on reconnection.
- **Lag handling**: When broadcast channel is full (256), slow subscribers receive Lagged(n) error. IM Bridge logs lag count; Server sends "missed N events" notification to WS clients.
- **Filtered subscription**: Subscribers process only relevant events via match at receiver side. Single channel, no need for multiple channels.
- **IM output throttling**: IM Bridge batches output at 5-second intervals. ANSI escape sequences stripped before sending.

## §3 — Plugin Protocol (IM + Tunnel)

### Communication Model

All plugins run as independent subprocesses communicating via stdin/stdout JSON-RPC 2.0. Any language that can read/write stdin/stdout can implement a plugin.

### IM Plugin Protocol

**Host -> Plugin (requests):**
- `initialize { config }` -> `{ capabilities: ["text","image"] }`
- `send_message { channel_id, text, reply_to? }`
- `send_image { channel_id, base64, caption? }`
- `shutdown {}`

**Plugin -> Host (notifications):**
- `on_message { channel_id, sender: { id, name }, text, is_command }`
- `on_status { status: "connected"|"disconnected"|"error", detail? }`

### Tunnel Plugin Protocol

**Host -> Plugin (requests):**
- `initialize { config }` -> `{ name, version }`
- `start { local_port, protocol }` -> `{ url }`
- `stop {}`
- `health {}` -> `{ status, uptime_secs }`

**Plugin -> Host (notifications):**
- `on_status { status: "ready"|"reconnecting"|"down", url?, reason? }`
- `on_metrics { bytes_in, bytes_out, connections_active }`

### Plugin Lifecycle State Machine

```
Pending -> Starting -> Ready
                |         |
           crash|    error|
                v         v
          Restarting   Failed
          (backoff)       |
                |    retry > 3
           retry <= 3     v
                |      Disabled
                v
            Starting
```

Backoff: 3s -> 6s -> 12s -> Disabled.

### Plugin Discovery

Plugins are discovered from `~/.rtb/plugins/` directory. Each plugin has a `plugin.toml` with metadata (name, type, version, command). Hot-loading via filesystem watcher (notify crate) — new plugin directories auto-load, removed directories auto-unload. No daemon restart needed.

## §4 — Session Persistence & History

### Storage Structure

```
~/.rtb/sessions/
├── index.json                # Session index (lightweight metadata)
├── {session_id}/             # nanoid 8-char
│   ├── meta.json             # Session metadata
│   └── events.jsonl          # Event stream (append-only)
```

### meta.json

```json
{
  "id": "a1b2c3d4",
  "name": "claude-dev",
  "type": "agent",
  "agent": { "provider": "claude-code", "model": "opus-4" },
  "cwd": "/home/user/project",
  "created_at": "2026-03-25T10:00:00Z",
  "last_active": "2026-03-25T14:30:00Z",
  "last_seq": 1842,
  "status": "running",
  "parent_id": null,
  "tags": ["work"]
}
```

### events.jsonl

Each line is one event with seq and timestamp:
```jsonl
{"seq":1,"t":"pty_out","ts":1711353600,"d":"$ ls\n"}
{"seq":2,"t":"agent_msg","ts":1711353602,"d":"I'll look...","streaming":true}
{"seq":3,"t":"tool_use","ts":1711353606,"tool":"Edit","input":{...}}
```

### Reconnection Protocol

1. Client reconnects with `last_seq` parameter
2. Server reads events.jsonl for seq > last_seq
3. Server sends `{"type":"replay","events":[...]}` batch
4. Server sends `{"type":"replay_done","last_seq":N}`
5. Switch to real-time Event Bus streaming

### Session Tree

`parent_id` field in meta.json enables hierarchical relationships. Fork an Agent conversation into a new branch. UI renders as collapsible tree list.

### Key Design Points

- **Append-only writes**: events.jsonl only appends. Crash loses at most the last (possibly incomplete) line. Startup validates and truncates corrupted last line.
- **Two-tier caching**: Hot data in memory ring buffer (last 5000 events). Cold data read from .jsonl on demand.
- **Auto-cleanup**: Configurable `max_session_age` (default 30 days) and `max_storage_size` (default 1GB). Background task scans and cleans expired sessions.
- **Output coalescing**: Consecutive pty_out events within 100ms window are merged into one before writing to jsonl, reducing disk I/O.

## §5 — Security Model

### Authentication

- 256-bit cryptographically random token generated on startup
- Token persisted to `~/.rtb/session.token` (0600 permissions)
- Token survives daemon restart unless manually rotated via `rtb token rotate`
- Three auth methods: query param (`?token=xxx`), header (`Authorization: Bearer xxx`), cookie (`rtb_token=xxx`, HttpOnly, SameSite=Strict)

### HTTP Security Headers

All responses include:
- `Content-Security-Policy` with per-request nonce for scripts
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `Permissions-Policy: camera=(), microphone=(), geolocation=()`
- `Strict-Transport-Security` (when accessed via tunnel)

### Rate Limiting (tower-governor)

| Endpoint | Limit |
|----------|-------|
| Auth endpoints (no token) | 5/min per IP |
| WebSocket connections | 10/min per IP |
| WebSocket messages | 100/sec per connection |
| Authenticated GET | 200/min per IP |
| Authenticated POST | 60/min per IP |

### IP Blocklist

- Auto-ban after 10 consecutive auth failures (15 min)
- Escalating bans on repeat offenses (up to 1 hour)
- Whitelist: 127.0.0.1/::1 always allowed, configurable CIDR whitelist
- Implementation: DashMap<IpAddr, BanEntry> with periodic cleanup

### Access Logging

Structured JSONLines audit log at `~/.rtb/logs/access.jsonl`. Includes HTTP requests, WS connections, auth failures, IP bans. Log rotation: 10MB per file, 5 files retained (~50MB). Implementation: tracing + tracing-appender.

### Defaults

- Binds to 127.0.0.1 by default (localhost only)
- Tunnel optional for remote access
- Token auto-generated, zero config needed
- All security headers on by default

## §9 — Smart Notifications & Task Pool

### Part A: Smart Notifications

#### Detection Engine — Three-Layer Signal Fusion

Replaces the current RTB's simple regex-based input-detector with a multi-signal confidence scoring system.

**Layer 1 — Process State Monitoring (highest weight, most reliable):**
- `tcgetpgrp(pty_fd)` tracks foreground process group. Shell becoming foreground = command completed. Pure OS-level signal, no output parsing needed.
- macOS: `proc_pidinfo(pid, PROC_PIDTASKINFO)` for CPU usage and thread state.
- Linux: `/proc/{pid}/stat` field 3 (state) + field 42 (wchan). `wchan = "wait_readable"` or `"do_select"` = waiting on stdin.
- Signals: `ForegroundChanged`, `ProcessWaitingOnStdin`, `ShellBecameForeground`, `ProcessExited`.

**Layer 2 — PTY Timing Analysis (auxiliary):**
- Output rate sliding window (5s): high-speed -> sudden silence vs always silent vs intermittent.
- Last output line structural features (not regex): ends_with_colon, ends_with_bracket, ends_with_question, has_cursor_at_end.
- Output-input alternation pattern detection (ping-pong = interactive program).
- Signals: `OutputStalled`, `OutputBurst`, `InteractivePattern`.

**Layer 3 — Output Semantic Analysis (supplementary):**
- Extensible tool signature library (`~/.rtb/signatures.toml`): cargo, npm, pytest, docker, git patterns for completed/error/input states.
- Shell prompt detection: configurable regex patterns, or auto-learn by recording first output line at startup.
- ANSI-aware: strip escape sequences before matching, but preserve cursor position info. Progress bar detection (\\r overwrite + percentage changes = running, not waiting).
- Signals: `ToolCompleted`, `ToolError`, `PromptInputDetected`, `ShellPromptDetected`.

**Signal Fusion:**
Weighted sum across all three layers. Each signal has a confidence weight (Layer 1 highest at 0.8-0.9, Layer 2 at 0.3-0.4, Layer 3 at 0.4-0.5). Trigger notification only when total confidence > threshold (default 0.7). Effectively prevents false positives (e.g., compile pause scores 0.3, below threshold).

**Graceful degradation:** If Layer 1 unavailable (permissions/platform), auto-fallback to Layer 2+3 with lowered threshold.

#### Notification Triggers

```rust
pub enum NotifyTrigger {
    ProcessExited { session_id, exit_code, command, duration },
    WaitingForInput { session_id, prompt_type, detected_text },
    LongRunningDone { session_id, command, duration, success },
    ErrorDetected { session_id, pattern, summary },
    AgentCompleted { session_id, summary },
    AgentNeedsApproval { session_id, action },
    AgentError { session_id, error },
}
```

#### Notification Channels

Multi-channel routing with per-trigger-type configuration:
- **Web Push**: WS push to browser, toast notification with action buttons
- **Desktop**: Tauri native notification, click to activate window and switch to session
- **IM**: Forward via IM plugins. Supports reply actions (e.g., `/approve` in Feishu)
- **Sound**: Web Audio API, configurable per trigger type

Configuration in `config.toml`:
```toml
[notification.rules]
process_exited   = { channels = ["web", "desktop"], min_duration = "30s" }
waiting_input    = { channels = ["web", "desktop", "im"] }
agent_completed  = { channels = ["web", "im"] }
agent_approval   = { channels = ["web", "desktop", "im"], urgent = true }
error_detected   = { channels = ["web", "desktop"] }
```

### Part B: Task Pool

#### Task Model

```rust
pub struct Task {
    id: TaskId,
    title: String,
    prompt: String,              // Full instruction for Agent
    priority: Priority,          // P0 | P1 | P2
    status: TaskStatus,          // Queued | Blocked | Running | NeedsReview | Completed | Failed | Cancelled
    target: TaskTarget,          // Agent { provider, model, cwd } | Command { cmd, cwd }
    depends_on: Vec<TaskId>,
    created_at: DateTime,
    started_at: Option<DateTime>,
    completed_at: Option<DateTime>,
    result: Option<TaskResult>,
}
```

#### Scheduling Strategy

1. **Priority-first**: P0 > P1 > P2, same priority FIFO.
2. **Dependency check**: All tasks in `depends_on` must be Completed. Otherwise status stays Blocked.
3. **Concurrency control**: `max_concurrent_tasks = 1` (default). Configurable, but Agent tasks recommended serial to avoid competing on same cwd.
4. **Auto-trigger**: When current Agent session becomes idle -> check task pool -> if executable task exists -> auto-create new session and execute.
5. **Review gate**: Task completion defaults to NeedsReview status. User confirms -> Completed, pick next. Configurable `auto_approve = true` to skip confirmation.

#### Multi-Entry Operation

- **CLI**: `rtb task add/list/pause/resume/cancel/reorder`
- **Web UI**: Expandable "Task Pool" panel in sidebar bottom. Drag-to-sort, click-to-edit, right-click menu. Running tasks show progress. Completed tasks show "Approve / Retry / Skip".
- **IM**: `/task add`, `/task list`, `/task approve`, `/task skip`

#### Persistence

Tasks persisted to `~/.rtb/tasks.jsonl`. Survives daemon restart.

#### Notification Integration

Task completion/failure triggers smart notifications. User can remotely approve via IM (`/task approve`) to continue the queue without opening the dashboard.

## §10 — Testing Strategy

### Test Pyramid

| Layer | Tool | Count Target | Scope |
|-------|------|-------------|-------|
| Unit tests | `cargo test` (rstest) | 200+ | core crate modules: buffer, store, acp_client, event_bus, detector, scheduler |
| Integration tests | `cargo test --test '*'` | 30-50 | Real PTY + WS, API endpoints, plugin lifecycle, Mock ACP server |
| Frontend component tests | Vitest + Testing Library | 50-100 | React components, hooks, utilities. Mock: msw |
| E2E tests | Playwright | 5-10 | Auth flow, terminal I/O, agent conversation, responsive layout |

### CI Pipeline (GitHub Actions)

**Stage 1 (parallel):** `cargo fmt --check`, `cargo clippy -D warnings`, `cargo test`, `bun test`

**Stage 2 (depends on Stage 1):** Integration tests, Playwright E2E

**Stage 3 (release tags only):** Build linux-x64/arm64, darwin-arm64 binaries. Build Tauri desktop (macOS dmg, Windows msi). Build Tauri mobile (Android apk, iOS ipa). Publish to GitHub Releases + Homebrew tap.

---

## Non-Goals

- Multi-user / multi-tenant support
- Enterprise SSO / LDAP integration
- Built-in code editor (use Agent or external editor)
- Web-based file manager
- Session recording / replay video export
